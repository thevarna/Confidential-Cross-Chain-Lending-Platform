# Confidential Cross-Chain Lending Platform (Final Corrected Hackathon Spec)

## 1. Product Overview and Purpose

Build a **Solana-native lending demo** for the Frontier track that combines:

- **Ika pre-alpha dWallet flows** for bridgeless authorization and cross-chain control primitives.
- **Encrypt pre-alpha encrypted-computation flows** for demonstrating loan-state logic and graph-based verification.

This is a **submission-safe prototype**, not a production privacy system. The official docs for both sponsor stacks explicitly state that they are **pre-alpha**:
- Encrypt pre-alpha has **no real encryption**; all data is public plaintext on-chain.
- Ika pre-alpha has **no real MPC signing**; signatures are generated by a single mock signer.

Therefore:
- do **not** use real user secrets,
- do **not** claim production confidentiality,
- do **not** claim production-grade bridgeless settlement,
- do **not** rely on current SDK behavior for security guarantees,
- do **not** present demo values as private production data.

The product should still look and feel like a serious institutional lending workflow, but the spec must describe it as a **demo-grade prototype** built around the sponsor primitives exactly as documented.

## 2. Hackathon Fit and Strategy

This project targets the Frontier track themes:

- **Bridgeless Capital Markets**
- **Encrypted Capital Markets**

The strongest first-prize angle is to present a **single, polished lending flow** that clearly shows both sponsor technologies in the critical path:

1. borrower creates a dWallet / authorization flow with Ika pre-alpha,
2. borrower creates a loan request using Encrypt pre-alpha graph logic,
3. lender funds the loan on Solana devnet,
4. borrower repays,
5. the system records the settlement outcome and dWallet lifecycle result.

The winning strategy is to keep the scope narrow and unmistakably sponsor-native. A smaller project that is fully implemented and well explained is stronger than a broad institutional platform that overclaims functionality.

## 3. Problem Statement

Blockchain liquidity is fragmented by chain, and lending workflows are hard to demonstrate credibly on-chain when private data is involved. Institutions also need a convincing workflow that shows how cross-chain control and confidential data handling could work.

This project addresses that by creating a **Solana-native lending prototype** that:

- uses **Ika pre-alpha dWallet instructions** to demonstrate bridgeless control over authorization and settlement workflow,
- uses **Encrypt pre-alpha graph execution** to demonstrate encrypted loan terms and arithmetic in a controlled, demo-safe setting,
- keeps the user experience focused on one clear lending path.

Because the sponsor stacks are pre-alpha, the spec treats privacy and settlement confidentiality as **product intent**, not as verified security guarantees.

## 4. Scope and Non-Goals

### In scope

- One collateral path for the demo.
- One loan currency for the demo, ideally USDC on Solana devnet.
- One lender per loan for the MVP.
- One borrower-to-lender loan lifecycle.
- One dWallet onboarding flow.
- One Encrypt graph / loan logic flow.
- A React frontend.
- Anchor-based Solana program.
- Devnet-only deployment.
- Synthetic or test-only values only.

### Out of scope

- Production privacy guarantees.
- Real sensitive borrower data.
- Multi-lender syndication.
- Secondary markets.
- Tokenized debt instruments.
- KYC / compliance workflows.
- Machine-learning credit scoring.
- Custom bridge implementation.
- Any claim that the current pre-alpha stack is final or production secure.
- Complex oracle-driven liquidation logic unless it is directly needed for a demo-safe MVP.

## 5. Target Users and User Roles

### Borrower
A user who creates a dWallet flow, submits a loan request, and repays the loan. The borrower uses a Solana wallet for transaction signing and a pre-alpha Ika flow for the dWallet portion.

### Lender
A Solana wallet holder who funds the loan and observes the loan lifecycle. The lender sees public demo fields and any redacted or synthetic summaries defined by the UI.

### Admin / Demo Operator
A minimal operational role used to configure collateral type, devnet addresses, test parameters, and pause/unpause functionality.

### Relayer / Automation Layer
A lightweight automated service or script that helps drive the pre-alpha flows and event indexing for the demo. This should be treated as infrastructure for the prototype, not as a trust assumption.

## 6. Core User Journeys

### 6.1 dWallet onboarding
The borrower connects a Solana wallet, then uses the official Ika pre-alpha TypeScript client to create or transfer dWallet authority using the documented instruction flow.

The UI should expose this as “Connect dWallet” and show the account / authority state relevant to the demo.

**Important correction:** the current Ika pre-alpha docs state there is no real MPC signing and that signatures are generated by a single mock signer. The spec must therefore describe this step as a pre-alpha authorization flow, not as production custody.

### 6.2 Loan request creation
The borrower enters demo loan terms such as amount, interest rate, and duration.

The frontend uses the Encrypt pre-alpha flow to compile a graph and prepare the encrypted-computation demonstration. The loan request is then written to Solana state.

**Important correction:** Encrypt pre-alpha documentation states there is no real encryption and that on-chain data is public plaintext. Therefore the demo must **never accept real sensitive data**. The UI should clearly label the fields as demo values and may redact or mask them in presentation, but the spec must not claim actual confidentiality on chain.

### 6.3 Loan listing and funding
The lender views active loan requests, sees the public summary fields required for the prototype, and funds the loan with the selected loan currency on Solana devnet.

The program updates loan state from Requested to Funded and emits the appropriate event.

### 6.4 Repayment and completion
The borrower repays principal plus interest before the due date. The program validates the repayment against the loan terms and updates the state to Repaid.

The demo should then show the final dWallet / settlement state transition relevant to the Ika flow, using the documented pre-alpha instruction set and test environment.

### 6.5 Default path
If the loan is not repaid before the deadline, the program moves the loan to Defaulted and exposes the lifecycle outcome in the UI. The default path should be simple and deterministic.

### 6.6 Account monitoring
The UI shows loan status, transaction history, event logs, and the dWallet / encrypted-flow state transitions needed for the demo. Anything sensitive must be synthetic, redacted, or demo-only.

## 7. What to Build / What Not to Build

### Build

### Do not build

## 8. Functional Requirements

### 8.1 User onboarding
- Borrower connects a Solana wallet.
- Borrower links the Ika pre-alpha dWallet flow.
- Borrower sees the created dWallet / authority state in the UI.
- The demo uses test or synthetic values only.

### 8.2 Loan request creation
- Borrower enters amount, rate, duration, and collateral type.
- Frontend prepares Encrypt graph input using the official pre-alpha client.
- Program stores the loan request on Solana and emits `LoanCreated`.
- UI marks the request as demo-only and must not imply real privacy guarantees.

### 8.3 Loan listing and browsing
- Frontend or an off-chain indexer/RPC query enumerates active loan accounts.
- UI shows public summary fields only.
- Any encrypted or privacy-sensitive presentation is treated as demo metadata, not a production privacy guarantee.

### 8.4 Loan funding
- Lender funds a selected loan.
- Program transfers the loan currency into escrow.
- Program updates state and emits `LoanFunded`.
- The demo shows the lender receipt and loan status update.

### 8.5 Loan repayment
- Borrower repays principal plus interest before the deadline.
- Program validates repayment and updates state to `Repaid`.
- The UI shows the successful completion path.

### 8.6 Loan default
- If the deadline passes without repayment, anyone allowed by the contract can trigger default handling.
- Program updates state to `Defaulted`.
- UI shows default outcome clearly.

### 8.7 Minimal governance
- Admin can register allowed collateral types and demo parameters.
- Admin can pause and unpause the loan creation flow.

### 8.8 Client-side handling
- The frontend uses the official Encrypt pre-alpha SDK and Ika pre-alpha TypeScript client as documented.
- The frontend never asks the user to enter real sensitive data.

## 9. Implementation Notes from the Official Docs

### 9.1 Ika pre-alpha client and instructions
Use the official Ika TypeScript client package:
- `@ika.xyz/pre-alpha-solana-client`
- built on `@solana/kit`

Use the documented Ika instruction flow and account derivations, including:
- `ApproveMessage`
- `TransferDWallet`
- `CommitDWallet`

The Ika docs also define the relevant pre-alpha environment and devnet/program setup. The spec should stay aligned with the documented pre-alpha client flow and should not invent a production MPC signing model.

### 9.2 Encrypt pre-alpha client and testing model
Use the official Encrypt pre-alpha packages and testing model:
- `@encrypt.xyz/pre-alpha-solana-client` for the TypeScript client
- `encrypt-types`
- `encrypt-dsl = { package = "encrypt-solana-dsl", ... }`
- `encrypt-anchor` for Anchor-based programs
- `encrypt-solana-test` for tests

Use the documented `#[encrypt_fn]` graph workflow, graph compilation, and mock testing model. The docs explicitly show:
- `run_mock(...)` for unit testing the graph,
- `parse_graph(...)` for graph-shape validation,
- devnet pre-alpha execution via the documented executor environment.

The spec must therefore treat Encrypt as a documented pre-alpha graph workflow, not as a general-purpose private execution platform.

## 15. API Contracts

### `initialize(config)`
Admin-only. Creates config, sets pause state, and sets allowed collateral and loan currency.

### `create_loan(params)`
Borrower-only. Creates a loan request and emits `LoanCreated`.

### `fund_loan(loan_id, amount)`
Lender-only. Funds the loan and emits `LoanFunded`.

### `repay_loan(loan_id, amount)`
Borrower-only. Repays the loan and emits `LoanRepaid`.

### `liquidate_loan(loan_id)`
Permissioned caller only, as defined by the contract. Marks default and emits `LoanDefaulted`.

### `cancel_loan(loan_id)`
Borrower-only and only before funding. Cancels the request.

### `update_config(...)`
Admin-only. Updates allowed collateral and demo settings.

### `pause()` / `unpause()`
Admin-only. Emergency controls.

## 16. Authentication and Authorization

- All on-chain actions require a valid Solana signature.
- Borrower actions require borrower authority.
- Lender actions require lender authority.
- Admin actions require admin authority.
- Ika pre-alpha actions must follow the documented instruction flow.
- The spec must not describe Ika pre-alpha as if it were a fully decentralized signing network.

## 17. State Management

### On-chain state
- Loan accounts are the source of truth.
- Status transitions are explicit.
- No off-chain database is required for core correctness.

### Frontend state
- Keep only session state and UI state locally.
- Fetch authoritative state from Solana accounts and event logs.

### Sponsor state
- Treat dWallet / encrypted-flow state as demo integration state, not as a production privacy source.

## 18. Error Handling

Use explicit errors such as:
- `LoanNotFound`
- `InvalidStatus`
- `NotAuthorized`
- `InsufficientFunds`
- `CollateralTypeNotAllowed`
- `LoanPaused`
- `DeadlinePassed`
- `InvalidRepaymentAmount`
- `DemoOnlyValueRejected`

The frontend must translate errors into human-readable messages.

## 19. Validation Rules

- All loan numbers must be positive.
- The collateral type must be whitelisted.
- The repayment amount must match the configured repayment logic.
- The deadline must be enforced by chain time.
- No real secrets may be submitted.
- Demo-only values must be clearly labeled.
- If the encrypted-flow or dWallet flow is unavailable, the demo should fail safely and clearly.

## 20. Edge Cases and Corner Scenarios

- **Partial funding:** not required for MVP.
- **Price volatility:** do not overbuild LTV logic unless strictly needed.
- **Endpoint failure:** show a clear retry path.
- **Missing sponsor endpoint:** fail gracefully and explain the dependency.
- **Key loss:** out of scope for MVP.
- **Replay or double-submit:** prevent with account state checks and nonces.
- **Version drift:** pin package versions to the official docs and note them in the README.
